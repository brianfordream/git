What is Version Control?

版本控制是一种记录文件内容变化，以便将来查阅特定版本修订情况的系统．它类似于一个文件数据库，可以在任意时间保存文件的快照，快照记录了文件变更的过程，在使用时可以将文件方便的还原到某一快照． 常见的版本控制分为集中式的版本控制系统，例如svn,分布式的版本控制系统，例如git.

集中式版本控制

集中式版本控制系统将完整的版本记录都保存在中央服务器上，使用者只获取到某个版本文件的拷贝,进行数据操作时需要与中央服务器通讯交换数据．集中式的缺点就是 单点故障，当服务器出现问题时，用户无法与服务器通讯提交更新，如果是磁盘损坏，更有可能丢失数据，最大的限制还是大部分操作都需要进行联网．

分布式版本控制

分布式版本控制系统中，客户端不只提取某一版本的文件快照，而是将整个代码仓库完整的镜像下来，这样，当中央服务器发生故障时， 使用任意一个镜像都可以恢复服务器.本地仓库包含了项目完整地提交历史，使用者可以在本地完成版本和分支操作，并可以提交代码到本地 仓库，当有网络时再提交给中央服务器．

What is Git

Git是目前最常用的分布式版本控制系统，由Linus设计用作Linux内核代码的管理．Git操作效率非常高，简单的移动指针就可以指向不同的分支和提交， Git鼓励使用分支开发代码，常用命令只有十几个，使用起来非常顺手． 下面对比下svn与Git的区别
SVN存储每个文件的变化
Git存储完整的文件快照

Git VS SVN

Git保存的是快照，SVN保存的是文件的变化.

Git几乎所有的操作都可以在本地完成，SVN需要与服务器通讯以获得老版本的文件．

Git对文件内容使用SHA-1 hash生成提交版本号，可以保证文件内容的完整性，而SVN使用递增的编号作为版本号．

Git工作流

git的三种状态

已修改(modified) → 工作区
↓ ↓
已暂存(staged) → 暂存区
↓ ↓
已修改(modified) → 仓库
已修改表示修改了文件，但是还没有打算提交． 已暂存表示将已修改的文件包含到下次提交的快照中． 已提交表示文件已经存在于本地仓库中，可以在分支上查看到提交历史． 在Git项目中与文件的三种状态对应的是三个文件区域的概念：工作区,暂存区以及Git仓库。

git的三个区域

工作目录是对项目的某个版本独立提取出来的内容。 这些从 Git 仓库的压缩数据库中提取出来的文件，放在磁盘上供你使用或修改。 暂存区域是一个文件，保存了下次将提交的文件列表信息，一般在 Git 仓库目录中。 有时候也被称作“索引”，不过一般说法还是叫暂存区域。 Git 仓库目录是 Git 用来保存项目的元数据和对象数据库的地方。 这是 Git 中最重要的部分，从其它计算机克隆仓库时，拷贝的就是这里的数据。 基本的 Git 工作流程如下： 在工作目录中修改文件。 暂存文件，将文件的快照放入暂存区域。 提交更新，找到暂存区域的文件，将快照永久性存储到 Git 仓库目录。 如果 Git 目录中保存着的特定版本文件，就属于已提交状态。 如果作了修改并已放入暂存区域，就属于已暂存状态。 如果自上次取出后，作了修改但还没有放到暂存区域，就是已修改状态

分支开发

在麦当劳点餐后，有人做汉堡，有人炸薯条，有人接可乐，最后，汇总到了前台,这是分支开发的一个现实缩影.当系统需要多人分工协作时，如果在一个 分支上工作，就需要频繁的合并，相互干扰，引入分支后，每个人在自己的分支上工作，在工作完成后进行合并操作.
